# 브레인스토밍 원본 대화 (요약)

본 파일은 meeting.md의 확정 사항이 도출된 원본 대화를 압축 보관한다. 세부 의사결정 과정을 추적하거나 추후 재논의 시 근거로 사용.

## 진행 흐름

1. **문제 제기** — 사용자: 리뷰어가 매 라운드 프로젝트 내용을 새로 읽어 토큰 소모가 누적된다.
2. **페인포인트 브레인스토밍** — 4가지 페인포인트 도출: 중복 읽기 / 기억 소실 / 전수 재검토 / Planner 재작성 비용. 사용자 지시: 전부 동등하게 해결 필요.
3. **전달 방식 전환 제안** — 사용자: 파일 기반 컨텍스트 전달로 전환.
4. **전체 재구상** — 사용자: "처음부터 다시 생각하자. Plan 단계만 본다. Planner는 writer로, 리뷰어와 팀을 구성해 직접 소통, 리뷰는 파일로 기록, 폴더 구조 `topics/{topic}/` 단일화, 합의되면 즉시 반영."
5. **Agent Teams 도입 결정** — 사용자 선택: Claude Code 실험 기능 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` 사용.
6. **컴팩션 문제 논의** — 장기 세션 컴팩션 완화 전략 S1~S5 중 S1(File-Truth) + S2(Phase-bounded) + S5(Delta-only) 채택.
7. **합의 우선 패러다임** — 사용자: "플랜 → 리뷰 → 합의 → 작성, 안된 부분 다시" 순서 확정.
8. **결정 단위 granularity** — B+C 하이브리드 채택 (섹션 스캐폴딩 + 결정 ID).
9. **번호 규칙** — 숫자만 사용. 특수문자(α/β/γ) 제거.
10. **라운드 상한** — 결정당 20회. 초과 시 사람 에스컬레이션.
11. **팀 분리** — Spec 팀과 Plan 팀은 별개로 띄운다 (컨텍스트 격리).
12. **브레인스토밍 파일명** — `meeting.md` 확정.
13. **plan.md 구조** — 섹션 내부에 결정 블록 임베드 (옵션 1).
14. **팀 소통 방식** — 팀원들이 자율적으로 대화하되 상태 태그 강제.
15. **meeting / design-doc 분리** — 브레인스토밍(백로그)과 topic-bound 설계를 별개 스킬로 분리.
16. **superpowers 스킬** — 별도 파일로 관리. meeting.md로 호출 결과 저장.
17. **세션 연속성** — meeting 직후는 자동 연결, 새 세션은 목록 제공.
18. **완료 판단** — 처음엔 `.done.md` 마커 제안 → 사용자 교정: `/work`까지 완료만 완료. `reflection.md`가 유일한 완료 마커.
19. **스크립트 자동화 범위** — 작업 현황 리스트 / 상태 판단만. 3개 .mjs 스크립트.
20. **스킬 이름** — `/design`이 아닌 `/design-doc` (미래 실제 design 스킬 대비).
21. **에이전트 파일명** — `design-doc-claude-reviewer.md` 등 AI 명시. Codex 불가 시 Opus 폴백 에이전트.
22. **폴더 구조** — flowness harness 구조에 맞춰 `harness/meetings/`, `harness/topics/` 배치.
23. **순서 반전** — 사용자 교정: spec이 먼저, plan이 나중. (WHAT → HOW)

## 핵심 사용자 발언 (원문 발췌)

- "지금 현재의 문제점 reviewer가 돌때마다 계속해서 프로젝트 내용, 해야할 내용을 라운드 진행할때마다 새로 read를 진행해 토큰소모를 너무나 가속시키는 현상이 있다."
- "전달 방식을 파일 기반으로 하는게 어떨까?"

- "δ 번호는 숫자로만하자 특수문자 알아보기 힘들어"
- "1 + 2 단계는 3회가아닌 최대 20회 까지 그 이후로는 사람이"

- "브레인스토밍 단계와 spec+plan 단계를 분리하자 meeting, design-doc 으로 분리하면 될듯"

- "범위는 작업 현황 (meeting, design-doc, working, done) 리스트 와 같은 기능들에 한에서만"
- "/design 으로 사용하지말고 /design-doc 으로 해줘 실제 디자인 스킬이 생길수도 있음"
- "flowness harness 구조에 맞춰서 폴더 생성하는게 좋을것 같음 그리고 agents/design-doc-reviewer.md 이 아닌 agents/design-doc-claude-reviewer.md 로 명시적으로 어떤 ai를 사용하는지 파일명에 나타내주면 좋음 codex를 사용할 수 없는 경우 다른 model (ex. opus, sonnet)을 이용하도록"

## 기각된 대안

- **기각**: 라운드마다 리뷰어 재생성 유지 → 토큰 문제 해결 못함.
- **기각**: 오케스트레이터 중계형 프로토콜 → 왕복 비용 + 팀 자율성 훼손.
- **기각**: Plan → Spec 순서 유지 → WHAT 미정 상태로 HOW 설계하는 위험.
- **기각**: 중간 단계 완료 마커 파일 (`spec.done.md` 등) → 상태 판단을 파일 존재/frontmatter로 일원화.
- **기각**: 과다한 스크립트 9종 제안 → 범위를 현황 조회 3종으로 축소.
- **기각**: `/plan` 스킬 유지 및 마이그레이션 제공 → 개인 플러그인이므로 breaking change 허용.
