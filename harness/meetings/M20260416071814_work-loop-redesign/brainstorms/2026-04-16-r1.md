# 브레인스토밍 r1: `/work` 루프 재설계

## 0. 배경 및 동기

사용자가 `/work` 스킬에서 세 가지 비효율을 동시에 지적:

1. **리뷰 사이클 비효율** — 매 라운드마다 5 Reviewer (rule/quality/security/performance/architecture) 병렬 실행 + aggregate. 작은 변경에도 6 에이전트 호출 × N 라운드 = 토큰 낭비, `code-review-r{N}[.{F}].md` 파일 누적.
2. **파일 기반 태스크 트래킹 부재** — 현재 `TaskCreate`/`TaskUpdate`(세션 메모리) 기반. 세션 종료 시 소실, 재개 시 파일 역추론 필요. `plan.md`의 `d-NNN`과도 분리.
3. **TDD/worktree 미성숙** — `internal-tdd`는 정의만 있고 강제 통합 없음 (Generator가 알아서 쓰길 기대). worktree 사용/관리 방식이 superpowers 등 타 플러그인 대비 사용성 낮음.

세 이슈는 얽혀 있음 — 리뷰 재설계 → 태스크 트래킹 모델에 영향 → TDD 단계와 매핑되어야 함. 따라서 통합 브레인스토밍.

**선택**: Q1에서 사용자가 "1" (단일 통합 미팅) 선택. 이슈를 분리하지 않고 한 번에 다룸.

## 1. 문제별 옵션 탐색

### 1.1 Q2. task 트래킹 모델 (입도)

| 옵션 | 입도 | 장점 | 단점 |
|------|------|------|------|
| A | d-NNN 단일 체크박스 | 단순 | TDD 5단계 진행 가시성 없음 |
| **B (선택)** | d-NNN × step 체크박스 (RED/GREEN/REFACTOR/COMMIT 등) | 중간 진행 추적 가능, plan.md와 매핑 | 파일 약간 길어짐 |
| C | d-NNN × file (변경 파일별) | 파일별 진행 추적 | plan.md 매핑 깨짐 |
| D | 자유형식 markdown | 유연 | 파싱/재개 어려움 |

**사용자 선택**: B. 이유: TDD 단계가 후에 강제될 가능성이 있고, 파일 기반 추적의 핵심 가치는 "어디까지 했는지 정확히" 확인하는 것이므로 step 단위가 적절.

### 1.2 Q3. TDD 적용 분기

| 옵션 | 형태 | 장점 | 단점 |
|------|------|------|------|
| **A (선택)** | `/work` (3-step) vs `/work-tdd` (5-step) 별도 스킬 | 명시적, 사용자가 의도적으로 선택 | DRY 위반 (공유 로직 복제) |
| B | 단일 `/work`에 mode 플래그 (`--tdd`) | DRY | 명시성 떨어짐, 잊기 쉬움 |
| C | plan.md frontmatter에 `tdd: true` 명시 | 토픽 단위 결정 (자연스러움) | 토픽 중간에 모드 변경 어려움 |
| D | 항상 TDD 강제 | 일관성 | UI prototype 등 TDD 부적절 케이스 차단 |

**사용자 선택**: A. 이유: "명시적으로 TDD를 사용하라" 라는 느낌이 좋음. 사용자가 슬래시 커맨드를 칠 때 의도가 분명히 드러남.

### 1.3 Q4. 리뷰어 구조

| 옵션 | 구조 | 장점 | 단점 |
|------|------|------|------|
| A | 현재 유지 (5명 병렬 + aggregate) | 다관점 | 토큰 낭비, 파일 누적 |
| B | 2명 직렬 (spec → quality) — superpowers 패턴 차용 | 균형 | 여전히 라운드별 누적 |
| **C (사용자 선택)** | **1명 (`code-reviewer`)**: 모듈화/최적화/시간복잡도/패턴 + lint + test 통과 | 최소 호출, 명확한 단일 게이트 | 다관점 손실 (security/performance 별도 강조 사라짐) |
| D | 리뷰 제거, lint/test만 | 최저 비용 | 휴먼 리뷰 가능 영역 누락 |

**사용자 결정**: C. 명시적으로 "work관련된 리뷰어들 다 지우고 code-reviewer로 검사한번 진행하면 되겠네". 검사 항목: 모듈화, 최적화, 시간복잡도, 각종 패턴, lint, test.

**삭제 대상 reviewer agents**: `rule-reviewer`, `quality-reviewer`, `security-reviewer`, `performance-reviewer`, `architecture-reviewer`. (단, `design-doc-*-reviewer` 3종은 design-doc 단계 전용이므로 별개로 유지.)

**리뷰 시점**: task.md의 모든 d-NNN × step이 체크된 후 1회. 라운드별 매번 호출이 아닌, "task.md 완료 마커" 시점.

### 1.4 Q5. Worktree

초기 옵션 (A: Flowness가 자동 생성/관리, B: 명시적 mode, C: 사용자 수동) → 사용자가 새 패러다임 제시:

> "워크트리는 하네스 워킹방식과 별개로 사용자가 선택하도록 하는게 좋겠다. 미팅을 하던 뭐를 하던 중간에 '워크트리 만들어줘' 하는 식으로 claude desktop code에서 워크트리 체크박스로 사용자가 직접 컨트롤하는것과 같은이치"

→ **Worktree 직교 원칙**: Worktree는 Flowness 워크플로우의 일부가 아닌, 사용자가 임의 시점에 "워크트리 만들어줘" 형태로 호출하는 별도 도구.

### 1.4.1 팀 단위 사용 시나리오 검증

사용자 질문: "좀 더 생각해봐야할게 팀단위로 사용한다면?? 문제가 없나?"

| 시나리오 | 직교 모델에서의 결과 |
|---------|--------------------|
| T1. A가 토픽 H1 시작, B가 토픽 H2 시작 — 같은 main에서 | 각자 직접 worktree 호출 가능. 충돌 없음. |
| T2. A 작업 중, B가 같은 토픽 합류 | A의 worktree 경로를 공유하거나 B가 별도 worktree 생성 (모두 사용자 결정). |
| T3. 토픽 중단 후 다른 작업으로 스위치 | git stash/branch switch가 아닌, 별도 worktree로 즉시 분기. |
| T4. CI 충돌 (두 토픽이 같은 파일 수정) | git merge 단계에서 처리 (Flowness 책임 밖). |
| T5. PR 베이스 브랜치 | `using-worktree` 스킬이 현재 브랜치/main 기반 결정. |

→ 팀 시나리오 모두 cleaner. Flowness가 worktree를 강제할수록 팀 워크플로우와 충돌 가능성 증가. **직교가 안전**.

### E3. code-reviewer 결과물 디렉토리

초기 제안 (reviews/ + review-tasks/ 분리) → 사용자가 통합 제시:

> "code-reviews 라는 폴더명 밑에 두고 테스크는 code-reviews/task-r1.md 로 하는건어때?"

→ **단일 `code-reviews/` 폴더**:

| 파일 | 역할 |
|------|------|
| `code-reviews/code-review-r{N}.md` | 라운드 N의 raw 리뷰 결과 (PASS/FAIL + 발견 사항) |
| `code-reviews/task-r{N}.md` | FAIL일 때 다음 라운드의 fix task 체크박스 |

라운드 N에서 FAIL → `task-r{N+1}.md` 생성 → /work이 이를 읽고 r{N+1} 진행 → 모두 체크 후 다시 code-reviewer 호출 → r{N+1} 결과 기록.

**사용자 결정**: 적용. 폴더명 단순화.

## 6. 의존성 / 트레이드오프 / 위험

### 6.1 상호 의존성

- **task.md ↔ TDD**: task.md의 step 칼럼이 TDD 5-step과 매핑되므로 두 토픽이 짝을 이룸.
- **review collapse ↔ task.md**: code-reviewer는 task.md 완료를 게이트로 호출되므로 task.md 먼저.
- **worktree 직교 ↔ /work**: /work이 worktree 호출 책임을 갖지 않음 → /work SKILL.md에서 worktree 호출 코드 제거 필요.

### 6.2 위험

| 위험 | 완화 |
|------|------|
| 5 reviewer 삭제 시 security/performance 사각지대 | code-reviewer prompt에 "보안/성능 명시 검증" 항목 포함 |
| /work과 /work-tdd 복사로 동기화 부담 | 변경 시 두 파일을 PR 체크리스트에 명시 |
| H/T 토픽 혼재 기간 (R4) | scripts/list-topics.mjs는 prefix 무관하게 작동하도록 검증 |
| task.md 첫 라운드 후 수동 편집한 내용 덮임 | 0단계에서 task.md 존재 시 skip + 사용자 경고 |
| AskUserQuestion이 비대화 환경(CI)에서 차단 | using-worktree에 `--auto-branch` 플래그 추가 옵션 |

### 6.3 비채택 옵션 — 왜 떨어졌나

- **2-reviewer 직렬 (Q4-B, superpowers 패턴)**: 사용자가 "1명으로 충분" 결정. spec/quality 분리는 design-doc 단계에서 이미 다관점 리뷰가 들어가므로 work 단계는 단일 게이트로.
- **단일 /work + --tdd 플래그 (Q3-B)**: 명시성 부족.
- **자동 worktree (Q5-A)**: 팀 시나리오 충돌 + 사용자 통제권 박탈.
- **R1 (rename 즉시 통합)**: 본 미팅 산출물에 무관한 큰 변경.

## 7. superpowers 비교 노트 (참고)

| 영역 | superpowers | Flowness 현재 | Flowness 신규 (본 결정) |
|------|-------------|---------------|----------------------|
| 태스크 파일 | plan.md (체크박스) | TaskUpdate (세션 메모리) | task.md (d-NNN × step) |
| TDD | Iron Law (강제) | Generator alaseo | /work-tdd 명시적 |
| 리뷰어 | spec → quality (2 직렬) | 5 병렬 + aggregate | code-reviewer 1 (단일 게이트) |
| Worktree | `.worktrees/` + check-ignore | `.flowness-worktrees/` 자동 | 사용자 호출, using-worktree |
| 서브에이전트 | 매 task fresh | 라운드 단위 fresh | 동일 |

**차용한 패턴**:
- 체크박스 markdown task 모델 (plan.md 영감)
- 사용자 검토 게이트
- worktree check-ignore 가드

**의도적 차이**:
- 단일 reviewer (Flowness는 design-doc에서 이미 다관점 리뷰 완료)
- TDD 옵션화 (UI prototype 케이스 고려)
